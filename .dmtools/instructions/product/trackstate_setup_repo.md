# TrackState Setup Repository

## Purpose

`trackstate-setup` is a **GitHub template repository** (`git@github.com:IstiN/trackstate-setup.git`) that serves as the fork-and-run starting point for TrackState.AI users. It is referenced as a Git submodule from the main `trackstate` repository.

When users want to deploy their own TrackState.AI instance, they fork `trackstate-setup`, enable GitHub Pages, and get a working Flutter SPA backed by their own Git repository — no server infrastructure required.

## Submodule location

```text
trackstate/
  trackstate-setup/       ← submodule: git@github.com:IstiN/trackstate-setup.git
  agents/                 ← submodule: git@github.com:IstiN/dmtools-agents.git
  .dmtools/
  .github/workflows/
  ...
```

## What the setup repository must contain

The setup repo is the user-facing deployment artifact. It must include:

### Required contents

- **Flutter web app build** — compiled SPA assets ready for GitHub Pages deployment
- **GitHub Pages configuration** — workflow or settings for automatic deployment from the default branch
- **Sample project data** — a pre-populated demo project with example Epics, Stories, Sub-tasks, comments, attachments, and workflow states so users see a working tracker immediately after forking
- **Project configuration** — default `project.json`, `config/` directory with issue types, statuses, workflows, fields, priorities, components, and versions
- **README** — step-by-step fork instructions, prerequisites, and first-use guide
- **Git LFS configuration** — `.gitattributes` tracking patterns for binary attachments
- **GitHub Actions workflow** — CI workflow to rebuild and redeploy the Flutter SPA when the user pushes changes

### Sample data structure (matching the candidate storage model from goal.md)


## Fork-and-run deployment model

The intended user flow is:

1. **Fork** — user forks `trackstate-setup` to their own GitHub account or organization
2. **Enable Pages** — user enables GitHub Pages (from the default branch or a `gh-pages` branch)
3. **Access** — user opens their GitHub Pages URL and sees the TrackState.AI Flutter app
4. **Authorize** — the app authenticates the user via GitHub OAuth/PAT and reads/writes tracker data from their forked repository
5. **Customize** — user modifies project configuration, issue types, workflows, and fields by editing JSON/MD files or through the app UI

No backend server is required. The Flutter SPA communicates directly with the GitHub API (or local Git commands in desktop mode).

## Agent guidance

When working on stories or tasks related to `trackstate-setup`:

- Agents are explicitly allowed to **modify files inside the `trackstate-setup/` submodule** when the task requires changes to the deployable template, demo data, Pages setup, or published app artifacts.
- Treat `trackstate-setup` as an actively maintained companion repository, not a read-only reference. If a change belongs in the deployable template, make it in `trackstate-setup` instead of forcing it into the main `trackstate` repo.
- When a task spans both repositories, agents should update both sides coherently: source/build logic in `trackstate`, deployable template/demo content in `trackstate-setup`.
- Keep repository boundaries clear: reusable source code, agent config, and build logic stay in `trackstate`; generated deployable assets, sample project content, and fork experience updates belong in `trackstate-setup`.
- The setup repo is the **user-facing product package** — treat it with the same quality bar as a shipped product
- Sample data must be realistic and demonstrate core features (hierarchy, JQL search, workflow transitions, comments, attachments)
- The fork experience must work end-to-end without manual setup steps beyond fork → enable Pages → open URL
- All sample issue metadata must follow the canonical frontmatter format defined in `goal.md`
- The Flutter SPA deployed from this repo must match the visual design defined in `DESIGN.md`
- Test the fork flow by verifying that a fresh clone of the setup repo serves the app correctly via a local web server
- Keep the setup repo lightweight — avoid committing build artifacts that can be generated by CI
- The setup repo should not contain the TrackState.AI source code; it contains the compiled app and sample project data
- Changes to the setup repo structure should be coordinated with changes to the main trackstate repository's storage model and CLI expectations

## Working agreement for multi-repo changes

If `trackstate-setup` is checked out and writable, agents may edit it directly as part of the same task. Those changes should be intentional, scoped to the setup repository's role, and reflected in any related instructions, workflows, or release process kept in the main `trackstate` repository. Agent post-actions are responsible for committing and pushing configured setup submodule changes before committing the updated submodule hash in the parent `trackstate` PR.

## Relationship to the main repository

| Repository | Role |
|-----------|------|
| `trackstate` | Source code, agents, CLI, Flutter app source, design system, CI/CD |
| `trackstate-setup` | Deployable template — compiled app + sample data, forked by users |
| `dmtools-agents` | Shared agent configs and scripts (submodule in `trackstate`) |

The main `trackstate` repo builds the Flutter app and publishes artifacts to `trackstate-setup`. Users never need to clone or build from `trackstate` directly — they only interact with their fork of `trackstate-setup`.
